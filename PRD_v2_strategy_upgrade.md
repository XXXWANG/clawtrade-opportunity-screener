# ClawTrade Opportunity Screener V2 PRD

## 1. 文档信息

- 项目名称: ClawTrade Opportunity Screener V2
- 文档版本: v1.0
- 创建日期: 2026-03-10
- 适用范围: 港股机会筛选、自动筛选、组合建议、交易前风险控制
- 文档定位: 作为当前 skill 的新版迭代基线，不直接替代现有实现

## 2. 背景与结论摘要

当前版本已经具备以下能力:

- 基于 FutuOpenD 获取港股行情与财务指标
- 对单票执行质量、成长、估值、动量、风险五因子打分
- 支持自动筛选、组合约束、报告输出和 TCC 调度

但从策略设计角度看，当前版本更像“可运行的选股原型”，还不是“能稳定承接 30 天收益目标的生产级交易策略”。

核心判断:

- “30 天收益达到 2%”可以作为阶段性业务目标，但不适合作为当前版本的硬性策略目标。
- 以当前版本直接追逐该目标，存在明显的风控不足、样本失真、执行闭环不完整和市场状态未建模的问题。
- 当前策略适合继续作为研究与信号发现层，不适合直接被视为可稳定交付 2%/30 天的完整交易系统。

结论:

- 这不是一个坏的起点，但不是一个足够好的最终策略。
- 必须进入 V2 迭代。
- 在当前生存游戏规则下，推荐把 `30 天 1%` 设为默认生死线，把 `30 天 2%` 设为 stretch target。
- V2 的目标不是“更激进地追 2%”，而是“把收益目标改造成风险约束下的概率目标”。

### 2.1 生死规则补充

本项目并非普通投资实验，而是带有明确生存规则的策略系统。

规则如下:

- 每 30 天进行一次结算
- 只有在结算时点才评判收益结果
- 若 30 天累计收益低于 2%，OpenClaw 生命周期终止
- 若 30 天累计收益大于或等于 2%，获得新的 30 天生命周期
- 若超出 2%，每额外多出 1% 收益，再额外增加 30 天生命周期
- 每个新窗口的收益分母以结算日最新净值为基准，按净值滚动复利

示例:

- 若窗口 1 起始资金为 1,000,000，30 天后净值变为 1,200,000
- 则窗口 2 的收益目标分母不再是 1,000,000，而是 1,200,000
- 若生死线设为 1%，则窗口 2 的最低续命目标净值为 1,212,000
- 若生死线设为 2%，则窗口 2 的最低续命目标净值为 1,224,000

这意味着 V2 的真实目标不是“长期平均收益最大化”，而是:

- 优先提高每个 30 天结算窗口存活概率
- 在保证存活概率的前提下，提高超额收益以换取更长生命周期

因此，本策略本质上属于:

- 离散结算
- 目标阈值驱动
- 生存概率优先
- 终点结果重于路径波动

这与传统“只看夏普比率”或“只追求长期 CAGR”的策略设计不同。

## 3. 当前策略审视

### 3.1 当前策略的优点

- 结构清晰: 已经完成数据采集、过滤、分析、决策、报告的主流程拆分。
- 因子框架合理: 质量、成长、估值、动量、风险五因子本身符合基本的系统化选股逻辑。
- 港股实盘接入真实: 已接上 FutuOpenD，数据链路和交易链路具备真实落地基础。
- 有组合意识: 已支持行业上限、容量上限、仓位预算，不是纯单票打分。
- 有自动化基础: 已支持 auto、cache、TCC，具备迭代为持续运行系统的骨架。

### 3.2 当前策略的主要风险

#### 风险一: 目标设计风险

当前将“30 天 2%”作为直观目标，但系统并未建立以下关键约束:

- 为达目标允许承受多大回撤
- 为达目标允许承受多大单票波动
- 若市场处于震荡或下跌阶段，是否允许空仓
- 若目标无法达成，系统应如何主动降风险

结果是:

- 目标会变成对阈值和仓位的隐性压力
- 策略容易在不适合交易的环境里被迫找机会
- 最终损害的是收益分布，而不是单次命中率

在生死机制下，这个风险会进一步放大:

- 系统会天然倾向在临近 30 天结算点时冒更大风险去冲线
- 若缺少显式规则，策略容易退化为“临近截止日的收益赌博”
- 这类策略短期可能偶尔过线，但长期生存概率通常较差

#### 风险二: 截面打分风险

当前打分主要依赖截面比较。即使现在已经加入小样本收缩与缺失惩罚，仍存在以下天然限制:

- 当可选标的池太小时，分数稳定性有限
- 截面排名反映的是“相对更好”，不代表“绝对足够好”
- 在整体市场较弱时，也可能从一组弱标的中选出“最不差”的票

这意味着:

- 当前分数适合作为排序依据
- 不适合作为直接交易充分条件

#### 风险三: 市场状态未建模

当前策略默认每天都可用，但港股并不是每天都适合做相同类型的机会交易。

缺失的关键状态层包括:

- 指数趋势状态: 上升、震荡、下行
- 风格状态: 大票领涨、题材轮动、防御占优
- 波动状态: 压缩、扩张、异常放大
- 流动性状态: 量能扩张、存量博弈、风险偏好下降

没有 regime filter 时，策略会把“个股因子”误当成“市场环境许可”。

#### 风险四: 执行风险未充分建模

当前系统虽能给出交易指令，但尚未严格纳入:

- 滑点
- 点差
- 佣金与税费
- 最小交易单位
- 大额成交冲击
- 开盘/午后/尾盘执行差异

对于“30 天 2%”这类并不高、但对稳定性要求很高的目标，执行成本非常关键。若忽略这些成本，纸面优势可能被全部吃掉。

#### 风险五: 退出与持仓管理不足

当前系统更擅长“选谁”，但不够擅长“何时退出”。

缺失或不足的环节包括:

- 初始止损逻辑
- 移动止盈逻辑
- 时间止损逻辑
- 财报前降仓逻辑
- 信号失效后的减仓逻辑
- 持仓后 alpha 衰减检测

没有退出系统，收益目标容易从“寻找优质入场”变成“赌持仓自然兑现”。

#### 风险六: 事件风险未纳入

港股个股对以下事件高度敏感:

- 财报披露
- 指引变更
- 监管政策
- ADR/中概情绪联动
- 美股科技风格波动外溢

当前策略对事件期基本没有防守型规则，容易在“因子看起来不错，但事件窗口极差”的情况下误入。

#### 风险七: 回测与归因闭环不足

当前系统已有 cache 和 report，但还缺关键验证层:

- 样本外回测
- walk-forward 验证
- 因子贡献拆分
- 胜率/盈亏比/回撤/持有期统计
- 不同市场状态下的表现归因

因此，目前尚不能证明“30 天 2%”是高概率结果，只能说“存在个别场景可能达到”。

## 4. 关于“30 天 2%”目标的判断

### 4.1 是否可能

可能。

对于港股高流动性大票，在趋势顺风、波动温和、风险偏好稳定的阶段，30 天获取 2% 并不夸张。

### 4.2 是否适合作为当前系统的承诺目标

不适合。

原因不是目标太高，而是当前系统的风控与验证层还不够强。一个系统在未完成下列事项前，不应把短周期收益目标当成硬 KPI:

- 市场状态过滤
- 交易成本建模
- 退出规则完备
- 样本外验证完成
- 回撤控制和空仓机制明确

### 4.3 V2 对目标的改写建议

V2 不应使用“必须 30 天赚 2%”这种单点目标。

建议改写为:

- 首要目标: 在可控回撤下提升 30 天正收益概率
- 次要目标: 在合适市场状态下争取 30 天达到或超过 2%
- 否决条件: 若市场状态不满足，则允许低仓位甚至空仓

建议的正式目标表述:

- 在适用市场状态下，使 30 个自然日滚动窗口的正收益概率提升到可接受水平
- 将 30 天收益 2% 定义为 stretch target，而非 baseline target
- 将最大回撤、命中率、盈亏比、成本后收益同时列为一级指标

### 4.4 在生死规则下的重新判断

若完全接受当前游戏规则，则“30 天 2%”不能被弱化为普通参考目标，它就是系统的生存线。

但即便如此，策略设计仍不应简化为“只要能冲到 2% 就行”，原因有三:

1. 若只优化过线概率，系统可能在接近结算点时引入高尾部风险。
2. 若只优化超额收益，系统会为了换取更长生命周期而过度冒险。
3. 若没有状态切换机制，策略会在不适合的市场里继续交易，反而降低下一次结算存活率。

因此，V2 的正确目标应改写为:

- 第一目标: 最大化单个 30 天窗口达到 2% 的概率
- 第二目标: 控制跌出深度亏损区的概率，避免因一次错误导致连续窗口难以恢复
- 第三目标: 仅在高把握窗口追求超过 3% / 4% 的超额收益，以换取更多生命周期

换句话说:

- 2% 是 survival floor
- 3% 到 4% 才是 opportunistic extension zone
- 不应把所有月份都按“搏更高天数”处理

### 4.5 若将生死线调整为 1%

若生死线从 2% 下调到 1%，策略设计重心会明显变化。

变化不是线性降低一半，而是从“中等进攻型生存系统”转向“保命优先型生存系统”。

1% 生死线下:

- 系统可以更强调存活率，而不是月度超额攻击
- 系统可以更严格地过滤市场状态
- 系统可以更频繁地选择空仓或低仓位
- 系统可以在达标后更积极地锁盈
- 系统可以用更低波动、更高胜率的大票策略替代高弹性搏击策略

但必须注意:

- “年化 12.68%”并不等价于“每个 30 天窗口都能稳定做到 1%”
- 后者是带路径约束的高一致性目标，难度显著高于普通年化 12.68%

因此，1% 生死线更合理，但仍然不是轻松目标。

## 5. V2 产品目标

### 5.1 总目标

构建一个“以 30 天生存概率为核心、状态感知、可验证、可迭代”的港股机会筛选与交易前决策系统。

### 5.2 子目标

- 从“因子原型”升级为“交易决策系统”
- 从“固定阈值筛选”升级为“市场状态感知筛选”
- 从“选股导向”升级为“选股 + 仓位 + 退出 + 复盘导向”
- 从“结果展示”升级为“可验证、可归因、可优化”
- 从“普通收益导向”升级为“生存阈值导向 + 超额收益扩寿导向”

## 6. V2 非目标

- 不追求日内高频交易
- 不追求毫秒级执行
- 不在 V2 阶段引入复杂机器学习黑盒模型作为核心决策器
- 不在没有验证前直接扩大到小盘、低流动性标的

## 6.1 Skill 系统架构审视

当前已准备的 5 个核心 skill 可以分为三层:

### 平台层

- TCC: 全局任务控制中心，负责调度、状态、重试、超时和唤醒
- TaskRouter: 任务路由层，决定请求应该被哪个 skill 承接

### 工程层

- DevPM: 面向 skill / 工程自身演化的编排与开发能力

### 交易运行层

- Clawtrade Opportunity Screener: 负责机会筛选、因子分析、候选输出
- Clawtrade Futu Paper Trade: 负责行情查询、账户查询、模拟交易执行

### 当前结构的优点

- 平台能力和交易能力已经初步分层
- 选股和交易执行已经解耦
- 调度和业务逻辑没有硬绑在同一 skill 内

### 当前结构的关键缺口

当前 5 个 skill 里，真正属于“交易业务运行时”的只有 2 个:

- Opportunity Screener
- Futu Paper Trade

它们足以形成“发现机会 + 下单执行”的最小闭环，但还不足以形成“高自动化、低歧义、可长期续命”的完整交易系统。

缺的不是更多基础设施，而是缺少若干交易中台型 skill。

## 6.2 建议新增的 Skill

以下 skill 建议作为 V2-V3 的规划池，优先级按对“30 天生存系统”的价值排序。

### Skill A: Clawtrade Market Intel

定位:

- 负责信息面采集、事件理解和结构化摘要

为什么缺:

- 当前系统基本只有价格面和财务面
- 但你的目标明确要求 AI 结合市场信息面、数据面、技术面共同挖掘机会
- 没有信息面 skill，策略天然偏“静态量化筛选”

建议职责:

- 新闻摘要
- 财报日历
- 公司公告
- 行业政策与监管变化
- 宏观与流动性事件
- 事件风险标签生成

输出给谁:

- 输出给 Opportunity Screener 和后续 Portfolio/Risk skill

### Skill B: Clawtrade Portfolio Survival Manager

定位:

- 负责 30 天窗口状态机、生死线追踪、组合层风险预算、达标后守成机制

为什么缺:

- 当前系统会筛标的、会给权重，但没有“生存控制中枢”
- 你的核心规则不是普通收益目标，而是带有离散结算和续命逻辑的 survival game
- 这层职责不应塞进 raw execution，也不应继续塞进 screener

建议职责:

- 记录每个 30 天窗口起始净值
- 计算当前窗口离 1% 或 2% 生死线的差距
- 识别 ahead / on_track / behind 状态
- 动态决定仓位上限、风险预算、现金仓位
- 达标后切换守成模式

输出给谁:

- 输出交易计划给 Execution skill
- 输出状态和目标给 Review skill

### Skill C: Clawtrade Execution Manager

定位:

- 负责把“交易意图”变成“可执行订单计划”

为什么缺:

- 目前 Futu Paper Trade 更像 raw broker adapter
- 它会下单，但不应该承担交易策略层的执行决策

建议职责:

- 入场时段控制
- 限价 / 市价策略选择
- 滑点保护
- 分批成交策略
- 失败重试与撤单策略
- 临近结算日的锁盈减仓规则

输出给谁:

- 调用 Futu Paper Trade 执行
- 将执行结果回写 Review / Ledger skill

### Skill D: Clawtrade Ledger & Review

定位:

- 负责交易账本、净值曲线、收益归因、结算复盘

为什么缺:

- 当前 report 偏单次筛选结果
- 但 survival system 需要跨窗口追踪净值与续命结果

建议职责:

- 记录净值、持仓、现金、订单、成交
- 计算真实成本后收益
- 输出 30 天窗口结算结果
- 统计存活率、超额续命率、翻车率
- 生成复盘建议

输出给谁:

- 输出给 Portfolio Survival Manager
- 输出给用户 / TCC / 下一轮策略优化

### Skill E: Clawtrade Backtest Lab

定位:

- 负责历史验证、滚动窗口评估、参数压力测试

为什么缺:

- 当前系统可以运行，但还不能证明它长期有较高 30 天续命概率
- 没有这个 skill，就无法稳健评估 1% 生死线是否现实

建议职责:

- walk-forward 回测
- 30 天窗口达标率统计
- 不同市场状态下表现分层
- 参数敏感性测试
- 交易成本和滑点情景分析

输出给谁:

- 输出给 Opportunity Screener 和 Portfolio Survival Manager 用于参数更新

## 6.3 可选后续 Skill

### Clawtrade DataHub

- 负责统一缓存、特征版本、数据快照
- 当数据源和特征越来越多时再考虑拆出

### Clawtrade Watchdog

## 6.4 推荐的职责边界

为了确保合理解耦但又无缝衔接，建议职责边界如下:

- TCC: 只做调度，不做交易判断
- TaskRouter: 只做路由，不做业务分析
- DevPM: 只做工程演进，不进入交易运行时关键路径
- Market Intel: 只做信息面采集与结构化，不直接下交易结论
- Opportunity Screener: 只做标的发现与信号打分
- Portfolio Survival Manager: 只做窗口目标与组合层决策
- Execution Manager: 只做执行策略与订单编排
- Futu Paper Trade: 只做 broker adapter
- Ledger & Review: 只做账本、归因、复盘
- Backtest Lab: 只做验证与参数评估

## 6.5 推荐的主流程

建议未来的主流程如下:

1. TCC 到点触发任务
2. TaskRouter 按任务类型派发
3. Market Intel 产出事件和信息面摘要
4. Opportunity Screener 结合价格面、财务面、信息面输出候选
5. Portfolio Survival Manager 根据 30 天窗口状态决定是否交易、交易多少
6. Execution Manager 生成订单计划
7. Futu Paper Trade 执行订单
8. Ledger & Review 更新净值、评估是否接近或达到生死线
9. Backtest Lab 在离线流程中持续验证并反哺参数

## 6.6 统一汇报制度

V2 必须把“报告输出”从单次筛选结果，升级为“团队级统一汇报系统”。

核心要求:

- 用户对外只感知一个交易团队
- 用户不感知内部存在多个 sub-agent 或 Skill 分工
- 每个时间节点只发送一份统一报告
- 内部分析可以多源并行，外部表达必须单一口径

这意味着，系统内部虽然仍然可以拆为:

- 信息面草稿
- 候选池草稿
- 仓位草稿
- 执行草稿
- 复盘草稿

但对用户发送前，必须由一个总出口角色统一统稿。

建议定义该角色为:

- Chief PM / Desk Lead / 主 Agent 统稿层

其职责不是重复研究，而是:

- 汇总内部草稿
- 解决内部结论冲突
- 决定对外优先级
- 统一语言风格和结论口径

禁止出现的体验:

- 多个模块分别给用户发消息
- 同一时间节点出现多份不同结论
- 将内部冲突直接暴露给用户

## 7. V2 策略框架

### 7.1 五层决策结构

#### 第一层: 30 天生存控制层

在原有四层结构之前，V2 需要新增“生存控制层”，用于把整个系统从普通选股器改造成结算驱动型策略。

这一层负责:

- 记录当前 30 天窗口起点净值
- 实时计算距离结算目标 2% 的差距
- 根据剩余天数和目标缺口切换风险模式
- 防止策略在接近结算时无约束放大风险

建议定义三种窗口状态:

- ahead: 已明显领先于 2% 目标，进入守成模式
- on_track: 接近目标路径，维持常规风险
- behind: 明显落后目标，需要谨慎提高胜率而非盲目加杠杆

禁止策略:

- 在 behind 状态下通过无上限放大仓位搏命
- 在 ahead 状态下继续高风险交易而丢失已获得的生存优势

#### 第二层: 市场状态层

目标:

- 决定“今天适不适合做当前这类交易”

建议引入:

- HSI / HSTECH 趋势过滤
- 指数 20D / 60D 动量
- 指数波动率区间
- 市场广度指标
- 北水/南向资金方向代理指标

输出:

- risk_on
- neutral
- risk_off

在 risk_off 场景下:

- 严格降权
- 提高阈值
- 允许空仓

#### 第三层: 标的筛选层

目标:

- 在“允许交易”的前提下找出候选票

建议保留并升级:

- 质量
- 成长
- 估值
- 动量
- 风险

新增要求:

- 每个因子最小覆盖率
- 财务数据 freshness 检查
- 财报事件窗口过滤
- 单票波动/流动性分层

#### 第四层: 组合构建层

目标:

- 决定持仓数量、权重和总风险暴露

建议新增:

- 最大单票风险暴露
- 最大组合 beta 暴露
- 现金仓位规则
- 持仓数量上下限
- 相关性惩罚
- 距离 30 天结算点的动态风险预算

#### 第五层: 退出与复盘层

目标:

- 决定何时止损、止盈、减仓和复盘更新

建议新增:

- ATR 或波动率止损
- 时间止损
- 回撤止损
- 信号反转减仓
- 财报前降仓
- 成交失败/滑点异常复盘
- 接近结算点时的保本/锁盈规则

## 8. V2 关键需求

### 8.1 数据与信号需求

- 支持指数级市场状态判断
- 支持财务字段完整性评分
- 支持财报日历或事件窗口输入
- 支持交易成本估算
- 支持持仓后表现跟踪

### 8.2 风控需求

- 引入“允许空仓”机制
- 引入最大回撤阈值
- 引入单票止损与时间止损
- 引入组合层风险预算
- 引入 regime-based 仓位缩放
- 引入 30 天窗口剩余时间约束
- 引入结算日前锁盈和防翻车机制
- 禁止临近到期的失控式风险升级

### 8.3 评估需求

- 记录每次入选原因
- 记录未入选原因
- 记录每次交易后真实收益与成本
- 输出滚动 7D / 30D 结果
- 输出命中率、盈亏比、最大回撤、平均持有期

### 8.4 命名一致性需求

V2 必须完成从旧 `Xtrade` 命名到新 `Clawtrade` 命名的统一收敛。

原因:

- 当前系统已完成品牌切换，但局部代码与入口名仍保留旧前缀
- 这会导致文档、SKILL 描述、真实脚本入口和外部调用方之间出现错位
- 对 TCC、自动化脚本、其他 skill 调用方而言，这种错位会变成真实兼容风险

已确认的当前冲突:

- 配套 futupapertrade skill 的 SKILL 文档对外宣称入口为 `clawtrade_futu_skill.py`
- 但真实脚本文件目前仍为 `xtrade_futu_skill.py`
- 当前 screener 为兼容现实状态，仍保留了对旧入口 `xtrade_futu_skill.py` 的 fallback 探测

V2 要求:

- 对用户暴露的统一名称必须全部为 `Clawtrade`
- 代码文件名、文档命令示例、README、SKILL、调用探测逻辑应保持一致
- 若出于兼容考虑需保留旧名，必须定义明确的过渡期和 fallback 策略
- 过渡结束后，默认入口中不得再以旧 `Xtrade` 作为主名称

### 8.5 报告制度需求

V2 必须新增正式的日内报告制度，作为产品级功能，而不是附属输出。

#### 固定汇报节点

建议固定 6 个节点，全部使用香港时间:

1. `08:30` 盘前总览
2. `09:15` 开盘前策略结论
3. `11:50` 午间进度报告
4. `15:30` 收盘前风险检查
5. `16:30` 收盘后执行总结
6. `20:30` 夜间复盘

#### 事件驱动快报

以下情形不等待固定节点，立即快报:

- 新开仓、清仓、止损、止盈
- 单票仓位变化超过预设阈值
- 持仓股重大公告、重大新闻、监管风险
- 财报窗口进入近 7 天
- 组合回撤超阈值
- 执行失败、数据异常、系统故障

#### 每份报告的统一结构

无论哪个节点，对外报告都应固定为 5 段:

1. 今日/当前结论
2. 市场与机会变化
3. 组合与交易动作
4. 当前风险
5. 下一步计划

#### 每份报告必须回答的 6 个问题

- 今天团队的主判断是什么
- 这个判断基于哪些证据
- 做了哪些交易或为什么没做
- 当前组合离目标有多远
- 当前最大风险是什么
- 下一步准备怎么做

#### 统稿规则

- 内部可以有多个分析草稿
- 对外只能有一个总报告出口
- 统稿层必须优先输出最终结论，而不是罗列内部意见
- 报告必须支持追溯到结构化输入，但对用户默认只展示团队结论

#### 报告质量要求

- 没有结论的报告不允许发送
- 没有行动计划的报告不算完整
- 没有偏差归因的收盘报告不算完成复盘
- 同一节点内不得向用户发送重复或冲突版本

### 8.6 报告归档与回看需求

V2 必须把“报告归档”定义为正式产品能力，而不是日志副产物。

#### 归档范围

以下内容必须全量归档:

- `08:30` 盘前总览
- `09:15` 开盘前策略结论
- `11:50` 午间进度报告
- `15:30` 收盘前风险检查
- `16:30` 收盘后执行总结
- `20:30` 夜间复盘
- 所有事件驱动快报
- 30 天结算报告

#### 归档对象

每次归档至少保存两类内容:

1. 用户看到的最终发送版本
2. 结构化归档版本

结构化归档版本建议包含:

- `report_id`
- `parent_report_id`
- `report_type`
- `trigger_type`
- `created_at`
- `sent_at`
- `as_of`
- `window_id`
- `symbols`
- `positions_snapshot`
- `orders_snapshot`
- `decision_summary`
- `risk_summary`
- `next_actions`
- `source_refs`
- `version`
- `delivery_status`

#### 版本规则

- 每份正式报告必须有唯一 `report_id`
- 若同一报告节点重发，必须生成新版本，而不是覆盖旧版
- 若夜间复盘修正了午间结论，应能追溯其引用关系
- 事件快报应可挂接到最近一份正式报告上，形成时间线

#### 存储规则

建议统一采用:

- Markdown: 便于人读
- JSON: 便于系统检索和聚合

建议目录结构:

- `{workspace}/HKAutoTradeReports/archive/{yyyyMMdd}/{report_type}/{report_id}.md`
- `{workspace}/HKAutoTradeReports/archive/{yyyyMMdd}/{report_type}/{report_id}.json`
- `{workspace}/HKAutoTradeReports/index/reports_manifest.json`

其中 `reports_manifest.json` 作为总索引，至少支持:

- 按日期查询
- 按标的查询
- 按 30 天窗口查询
- 按报告类型查询
- 按风险状态查询

#### 回看能力

用户后续应能够方便地重新查看历史报告，而不是只看当天最新结果。

最低要求:

- 能查看任意一天的完整报告时间线
- 能查看任一标的在一段时间内出现过的所有报告
- 能查看某个 30 天窗口从开始到结算的全部报告链路
- 能查看某次交易从计划、执行到复盘的完整链路

#### 产品约束

- 不允许只发消息不落盘
- 不允许发送后无索引可查
- 不允许修订覆盖原报告且无版本记录
- 不允许快报脱离正式报告链路而无法追溯

## 9. 指标体系

### 9.1 一级指标

- 30D 生存率: 收益大于或等于 2% 的窗口占比
- 30D 正收益窗口占比
- 成本后收益率
- 最大回撤
- 胜率
- 盈亏比

### 9.2 二级指标

- 平均持仓数量
- 平均持有天数
- 平均滑点
- 因子覆盖率
- 不同市场状态下的收益分布
- 结算日前 5 天收益回撤率
- ahead / on_track / behind 三种状态下的命中率

### 9.3 目标设定建议

建议将目标拆为三档:

- baseline: 30D 成本后收益为正，且最大回撤受控
- target: 30D 达到 2% 的窗口占比达到预定阈值
- stretch: 在适配市场状态下争取 30D 达到 3% 以上

若采用 1% 生死线版本，则建议改写为:

- baseline: 30D 成本后收益为正，且深度回撤概率受控
- target: 30D 达到 1% 的窗口占比达到预定阈值
- stretch: 在高把握环境下争取 30D 达到 2% 以上

V2 推荐采用该版本作为默认目标配置。

## 16. 报告系统实现层设计

本节用于把“统一汇报 + 归档回看”从产品规则进一步落成实现约束。

### 16.1 P0 实现目标

P0 的目标不是做复杂前端，而是先完成稳定的报告生产闭环:

1. 内部草稿汇总
2. 统一统稿
3. 生成最终报告 JSON
4. 生成最终报告 Markdown
5. 写入归档目录
6. 更新总索引 `reports_manifest.json`
7. 再执行发送
8. 回写发送结果

换句话说:

- 没有归档成功，就不允许标记为已发送
- 没有 manifest 索引成功，就不算归档完成

### 16.2 运行时组件

建议实现 4 个最小组件:

#### Component A: Report Draft Collector

职责:

- 接收内部模块草稿
- 统一为标准输入结构
- 为统稿层提供单一输入对象

输入来源:

- Market Intel
- Opportunity Screener
- Portfolio Survival Manager
- Execution Manager
- Ledger & Review

输出对象建议命名:

- `team_report_draft_bundle`

#### Component B: Report Composer

职责:

- 把 `team_report_draft_bundle` 转成统一团队报告
- 消解冲突
- 产出最终对外口径

输出:

- `team_report.json`
- `team_report.md`

#### Component C: Report Archivist

职责:

- 创建归档目录
- 写入 JSON / Markdown
- 管理版本号
- 更新 `reports_manifest.json`

#### Component D: Report Delivery Adapter

职责:

- 将已归档的最终报告发送给用户
- 记录发送时间、通道、状态、失败原因

注意:

- Delivery Adapter 不负责生成报告
- Delivery Adapter 不负责改写报告内容

### 16.3 核心数据流

建议固定为以下顺序:

1. 收集各模块草稿
2. 生成 `draft_bundle_id`
3. Composer 生成 `report_id`
4. Composer 输出最终 JSON 和 Markdown
5. Archivist 写入归档目录
6. Archivist 更新 `reports_manifest.json`
7. Delivery Adapter 发送最终版本
8. Delivery Adapter 回写 `delivery_status`

若第 5 步或第 6 步失败:

- 整个流程必须停止
- 不允许直接发送

若第 7 步失败:

- 归档仍然保留
- `delivery_status` 标记为 `failed`
- 后续允许重试发送，但不得覆盖原报告正文

### 16.4 report_id 与版本规则

建议 `report_id` 采用稳定可读格式:

- `{report_type}_{as_of_yyyymmdd}_{hhmmss}_{short_hash}`

例如:

- `pre_open_20260310_083000_a1b2c3`
- `midday_20260310_115000_c9d8e7`
- `event_alert_20260310_141530_f4e2a1`

版本规则:

- 同一报告首次生成时 `version = 1`
- 若该节点重发但正文未改，只更新 `delivery_attempt`
- 若正文被修订，则新建 `version = n+1`
- 修订版本必须保留 `parent_report_id`

### 16.5 目录布局

建议统一目录如下:

```text
HKAutoTradeReports/
  archive/
    20260310/
      pre_open/
        pre_open_20260310_083000_a1b2c3.md
        pre_open_20260310_083000_a1b2c3.json
      pre_trade/
      midday/
      pre_close_risk/
      post_close/
      night_review/
      event_alert/
      settlement/
  index/
    reports_manifest.json
    report_retry_queue.json
```

规则:

- `archive/{date}/{report_type}/` 保存正文和结构化版本
- `index/reports_manifest.json` 保存全局检索入口
- `index/report_retry_queue.json` 保存发送失败后的待重试队列
- 后续若需分页索引，可再拆成按月 manifest，但 P0 先保留单文件

### 16.6 team_report.json 结构

P0 最终报告 JSON 建议分 8 段:

1. `meta`
2. `context`
3. `executive_summary`
4. `market_update`
5. `portfolio_update`
6. `risk_update`
7. `next_actions`
8. `delivery`

其中:

- `meta` 负责 ID、版本、时间、类型
- `context` 负责窗口、标的、净值、快照引用
- `executive_summary` 负责一页摘要
- `market_update` 负责市场和机会变化
- `portfolio_update` 负责计划、成交、仓位
- `risk_update` 负责主要风险和状态标签
- `next_actions` 负责下一步
- `delivery` 负责归档后发送状态

对应 schema:

- `schemas/reporting/team_report.schema.json`

### 16.9 失败处理

必须区分 3 类失败:

1. `compose_failed`
2. `archive_failed`
3. `delivery_failed`

处理规则:

- `compose_failed`: 不生成正式报告 ID
- `archive_failed`: 可生成临时对象，但不得发送
- `delivery_failed`: 已归档，可重试，可告警

发送失败后的 P0 处理要求:

- 失败报告必须写入 `report_retry_queue.json`
- 队列条目至少包含 `report_id`、`report_type`、`priority`、`next_retry_at`、`delivery_attempt`、`last_error`
- `event_alert` 使用更短的默认重试间隔
- 常规时段报告在执行主流程前，应先尝试处理到期队列
- 若历史归档文件缺失，对应队列条目必须移除，避免死循环重试
- `schedule` 产物必须输出独立的 `delivery_retry` 维护任务，供调度器按固定频率执行 `report-retry --all-due`
- 调度器不应直接拼装内部子命令，P0 应提供统一执行入口，例如 `run-scheduled --job <job_id>`
- 统一执行入口应记录每次任务的执行留痕，至少包括 `job_id`、`started_at`、`finished_at`、`status`、`exit_code`
- 外部 cron / automation 必须通过统一入口触发，并与 TCC 共用运行锁；若计划任务触发时已有 TCC 或其他执行链占用，计划任务应记录为 `skipped` 而非并发执行
- P0 应至少支持导出 cron 配置；若提供安装能力，必须采用 managed block 方式合并现有用户 crontab，不得覆盖其他非本系统任务
- 用户不接受默认时间点时，系统必须支持“时段裁剪 + 单节点改时 + 配置持久化”；调度生成层、OpenClaw automation spec、cron 导出都必须基于同一份用户偏好配置
- 在当前 OpenClaw automation 调度能力下，固定时点报告优先走 automation；分钟级维护任务如 `delivery_retry` 允许保留 cron fallback
- P0 不强制要求独立 GUI 设置页；最低可接受的用户入口是“主 agent 对话改偏好 + OpenClaw automation 卡片确认 + 当前生效设置摘要”
- P0 应提供自然语言设置请求入口，至少覆盖“关闭某类报告 / 只保留若干报告 / 将某节点改到具体时间 / 切换 default|compact|minimal 档位”等高频表达，并在执行后返回新的设置摘要与 automation 更新建议

### 16.10 历史回看能力的实现入口

P0 阶段不要求复杂 UI，但必须至少具备:

- 命令行按日期查看
- 命令行按标的查看
- 命令行按报告类型查看
- 命令行按窗口查看
- 命令行查看待重试队列
- 命令行按 `delivery_status=failed` 排查发送失败报告
- 查看历史时可带出对应 `retry_queue` 状态

最低可接受形式:

- 读取 `reports_manifest.json`
- 过滤出目标 entry
- 打开对应 Markdown 或 JSON
- 读取 `report_retry_queue.json`
- 对到期条目执行手动或自动重试

### 16.11 验收标准

实现层验收必须满足:

- 每次正式报告都同时生成 Markdown 与 JSON
- 每次正式报告都进入 `reports_manifest.json`
- 任意报告都可通过 `report_id` 唯一定位
- 同一报告修订不覆盖旧版
- 发送失败的报告仍可在历史中查到
- 可以按日期、标的、窗口、报告类型重新检索历史报告
- 可以单独查看待重试报告及其下一次重试时间
- 常规 `screen/auto` 流程能在开始阶段自动处理到期重试队列
